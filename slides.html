The Three Currencies of ML Performance

Compute · Memory · Communication — one mental model for ML training performance

1 / 26

What does "performance" mean?

  • Performance = throughput — useful work per second (tokens/s, samples/s)
  • Yardstick = MFU (Model FLOPs Utilization) = achieved FLOPs ÷ peak FLOP/s
  • Useful FLOPs of a dense transformer ≈ 6·N·D (2·ND forward + 4·ND backward)
  • A GPU is, to first order, a matmul machine → performance = keep the tensor cores fed
2 / 26

One question, three answers

  • Ask only: what is starving the tensor cores right now?
  • ① Compute & bandwidth — the direct lever (the roofline)
  • ② Memory capacity (VRAM) — indirect & double-edged
  • ③ Communication — dominates at scale
  • One budget (keep cores fed), three currencies — every technique is a trade between them

§1 Compute & Bandwidth

the roofline — the direct lever

4 / 26

Counting the FLOPs

  • A matmul (M×K)·(K×N) costs 2·M·N·K FLOPs (multiply-accumulate = 2)
  • Chained up → transformer training ≈ 6·N·D = the numerator of MFU
  • FLOPs (work done) ≠ FLOP/s (rate the hardware sustains); MFU is the ratio
5 / 26

Q1 — Are your matmuls running at peak?

  • Compute intensity: bigger matmuls saturate the tensor cores → more achieved FLOP/s
  • Tiling & wave quantization: use round sizes (multiples of 64/128, powers of two) so the last wave isn't half-empty
  • Lower precision: BF16 ≈ a free lunch for training; FP8 for large LMs (DeepSeek-V3, <0.25% loss vs BF16, needs Hopper+)
6 / 26

Q2 — Are you even compute-bound? (the roofline)

  • Set by arithmetic intensity = FLOPs per byte, vs the roofline ridge point
  • Big matmuls (QKV, MLP, attn output) → compute-bound
  • softmax / LayerNorm / GELU / dropout / residual → memory-bound (limited by HBM bandwidth)
  • HBM: A100 ≈ 2 TB/s, H100 ≈ 3.35 TB/s — most "surprising" slowness is a memory-bound op
7 / 26

Kernel fusion (torch.compile or hand-written kernels)

  • Memory-bound ops round-trip HBM per op; fuse the chain into one kernel → intermediates stay on-chip
  • Get it automatically from torch.compile (TorchInductor → Triton), or hand-write in Triton / CUDA
  • Launch-bound / GPU-starved: CPU can't launch kernels fast enough → CUDA graphs replay the launches
8 / 26

FlashAttention — the same trick, on attention

  • Not a cheaper algorithm — FLOPs stay O(S²), result bit-for-bit exact
  • ① Fuse the whole attention (QK^T → softmax → ×V) into one kernel → S×S never written to HBM
  • ② Tile into SRAM-sized blocks (online softmax) → memory footprint O(S²) → O(S)
  • A pure memory / bandwidth win — the cleanest proof attention was memory-bound, not FLOP-bound
9 / 26

Quantization — not a free lunch

  • Weight-only (W4A16, GPTQ, AWQ): weights low, activations 16-bit → dequant first; saves bytes, not compute
  • Native low precision (W8A8 / FP8 both operands): runs on FP8 tensor cores → ~2× FLOP/s
  • ✅ Memory-bound (decode): weight-only is a big win (fewer bytes)
  • ⚠️ Compute-bound (training/prefill): weight-only can be slower (dequant overhead, no compute saving)
  • Needs native HW: H100 / L40S have FP8, A100 does not (falls back to BF16 + dequant)

§2 VRAM

capacity ⇄ throughput — indirect & double-edged

11 / 26

VRAM — the capacity ⇄ throughput balance

  • Not a speed knob — you either fit, or you OOM
  • Headroom helps: spare VRAM → bigger batch/seq → higher arithmetic intensity → higher MFU
  • OOM hurts: to fit, you pay another currency — compute or communication
  • Every memory-saving technique spends a currency
12 / 26

What actually eats your VRAM

  • Model states = params + gradients + optimizer states (≈ independent of batch/seq)
  • Residual states = activations + temp buffers + fragmentation
  • Memory is dynamic: the forward–backward peak towers over the static model
13 / 26

16 bytes/param — and it's just AMP

  • 2B BF16 params + 2B BF16 grads + 4B FP32 master weights + 4B Adam m + 4B Adam v
  • ⚠️ Mixed precision does not save model-state memory — same 16 B/param as pure FP32
  • AMP's real win is faster compute (BF16/FP8 tensor cores ~2× FP32); it only trims activation memory
14 / 26

Activations — the elephant in the room

  • The memory everyone forgets — and it explodes with batch × sequence length
  • Almost negligible for short sequences; from ~2–4k tokens it takes off
  • At long context / large batch: by far the largest memory burden, while model states stay constant
  • A model that fits at 2k context can OOM at 32k — nothing about the weights changed
15 / 26

Activation checkpointing — and how to be selective

  • What: drop activations in forward, recompute them in backward (trade memory for compute)
  • Full recomputation ≈ +30–40% compute; selective is far cheaper
  • Selective rule: recompute the core attention (huge S×S activations, few FLOPs); store the expensive feedforward matmuls
  • Decided by cost to regenerate, not speed — recompute the cheap, store the expensive
16 / 26

The rest of the VRAM toolkit

  • Gradient accumulation: big effective batch, only one micro-batch of activations resident
  • Parallelism (ZeRO/FSDP, TP, PP, SP/CP): shard across GPUs → pay communication for capacity
  • CPU/NVMe offload: moves data over PCIe (no recompute) — slower than checkpointing; a last resort
  • Fragmentation: not OOM, just no contiguous block → expandable_segments (safest on a single GPU)

§3 Communication

the bandwidth hierarchy — direct at scale

18 / 26

The bandwidth hierarchy

  • SRAM ↔ HBM ↔ NVLink (GPU↔GPU) ↔ PCIe (CPU↔GPU) ↔ InfiniBand/EFA (node↔node) ↔ storage
  • Each rung down ≈ 10× slower
  • Universal trick: overlap communication with compute → it becomes "free"
  • Failure mode: exposed communication on the critical path → cores idle
19 / 26

CPU → GPU: feeding the data pipeline

  • CPU and GPU run asynchronously — the CPU launches kernels onto a queue and races ahead
  • Why Python being slow rarely matters: a GPU step dwarfs the Python that dispatched it (launch overhead hides behind compute)
  • Pinned memory + non_blocking=True → DMA overlaps the host→device copy with compute
  • num_workers / prefetch_factor → load + augment the next batch while the GPU computes the current one
  • Disk → VRAM directly = GPUDirect Storage (needs nvidia-fs + a supported filesystem; not universal)
20 / 26

GPU → GPU: parallelism (TP / SP / PP)

  • Tensor parallel (TP): split a layer's matmul — use when a single layer doesn't fit and the interconnect is fast (intra-node, TP ≤ 8)
  • Sequence parallel (SP): shard the leftover LayerNorm / dropout / residual along the sequence — for long-context activation memory
  • Pipeline parallel (PP): split layers into stages — tolerates slow links, but fiddly (manual balancing + pipeline bubbles)
21 / 26

FSDP / ZeRO — the default

  • Plain data parallelism replicates the whole model on every GPU (~120 GB/GPU for 7.5B) — almost all redundant
  • ZeRO shards model states across DP ranks, rebuilt just-in-time: ZeRO-1 optimizer, 2 +gradients, 3 (= FSDP) +params
  • Nearly free: ZeRO-1/2 move the same bytes as plain DP; ZeRO-3 overlaps its comm with compute
  • Your default for multi-GPU memory savings — start here, add stages (then TP/PP) as needed

Every technique is a trade

  • Activation checkpointing — spend compute to buy memory
  • ZeRO / FSDP / offload — spend communication to buy memory
  • Bigger batch — spend memory to buy compute (higher MFU)
  • FlashAttention / kernel fusion — exact, same-FLOPs free lunches (quantization joins only on the right hardware)
  • Loop: measure which currency you're short on, then spend a cheaper one
